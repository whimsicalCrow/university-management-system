# Architecture Memlog — Thesis Collaboration Portal MVP

**Scope:** MVP phase architecture spine for thesis collaboration portal  
**Purpose:** Foundation for end-to-end feature delivery by graduation (2026-07-31)  
**Altitude:** Initiative (spans all 8 MVP features)  
**Date:** 2026-06-26

---

## Decisions Logged

### AD-1: Paradigm — Clean Architecture with CQRS
**Type:** decision  
**Binds:** Domain layer owns business rules (Student, Professor, Thesis, Update); Application layer orchestrates via MediatR; Infrastructure provides persistence  
**Prevents:** Anemic models, logic scattered across controllers, tight coupling to EF Core  
**Rule:** Every feature flows through Query/Command handlers; domain entities remain persistence-agnostic  
**Status:** [ADOPTED] — Already installed in .csproj (MediatR, FluentValidation packages)

### AD-2: Presentation Layer — Blazor Server
**Type:** decision  
**Binds:** All UI pages (Login, Dashboard, StudentDashboard, etc.) are Razor components; no separate frontend build  
**Prevents:** Complex SPA tooling, frontend/backend coordination overhead  
**Rule:** Razor pages call MediatR handlers via async Tasks; authorization via ASP.NET Core Identity claims  
**Status:** [ADOPTED] — Already in use for all 8 pages

### AD-3: Authentication — ASP.NET Core Identity
**Type:** decision  
**Binds:** Replace UserSessionService with ASP.NET Core Identity + database persistence  
**Prevents:** In-memory sessions lost on restart, no multi-device support, no role hierarchy  
**Rule:** Users → AspNetUsers table; roles (Student/Professor) → AspNetRoles; auth via Claims  
**Migrates From:** UserSessionService (in-memory enum-based)  
**Status:** [NEW] — To implement Week 2

### AD-4: Data Persistence — SQL Server + Entity Framework Core
**Type:** decision  
**Binds:** EF Core Code-First migrations generate SQL Server schema from domain models  
**Prevents:** Ad hoc SQL queries, schema drift, no audit trail of migrations  
**Rule:** DbContext owns all model → database mapping; migrations committed to repo  
**Deployment:** SQL Server 2022 in Docker; connection string via environment variable  
**Status:** [NEW] — To implement Week 1

### AD-5: CQRS Handler Pattern — MediatR
**Type:** decision  
**Binds:** Each feature (F1–F8) splits into Command (create/update) or Query (read); handler validates, calls domain, persists  
**Prevents:** Monolithic controllers, business logic in views  
**Rule:** Command handlers: validate input → call domain method → persist → return result; Queries: fetch + return DTO  
**Tooling:** MediatR v11.1.0 (already installed), FluentValidation v11.9.0  
**Status:** [ADOPTED] — Infrastructure present, no handlers written yet

### AD-6: Dependency Injection — Microsoft.Extensions.DependencyInjection
**Type:** decision  
**Binds:** Program.cs registers services (DbContext, repositories, handlers, SignalR hubs); constructor injection throughout  
**Prevents:** Service Locator anti-pattern, global singletons  
**Rule:** MediatR handlers, repositories, and domain services registered in DI container; Blazor components receive via `@inject`  
**Status:** [ADOPTED] — ASP.NET Core standard

### AD-7: Authorization — Role-Based Access Control (RBAC)
**Type:** decision  
**Binds:** Two roles: Student, Professor; access policies defined in Program.cs; Razor pages/MediatR handlers check claims  
**Prevents:** Data leakage between roles, unauthorized updates  
**Rule:** MainLayout redirects unauthenticated → Login; professor page routes require `Professor` role; student routes require `Student` role  
**Scope MVP:** No admin role; professors only self-assign  
**Status:** [NEW] — To implement Week 2

### AD-8: Database Schema — Invariants
**Type:** decision  
**Binds:** Identity tables (AspNetUsers, AspNetRoles, AspNetUserRoles) + Domain tables (Students, Professors, Assignments, ThesisUpdates, Feedback)  
**Prevents:** Orphaned records, role/user sync issues  
**Rule:** Foreign key constraints enforced; cascade deletes on Student → Updates/Feedback; soft-delete for audit (Phase 2)  
**Scope MVP:** No attachment storage; text-only updates  
**Status:** [NEW] — Design in Week 1; migrations auto-generate

### AD-9: Layered Folder Structure
**Type:** decision  
**Binds:** 
  - University.Web: Controllers (if used), Views (Razor), Components, Services (SignalR hubs, session managers)  
  - University.Application: Commands, Queries, Handlers, Validators, Dtos  
  - University.Domain: Entities, Aggregates, Value Objects, Domain Services, Repositories (interfaces only)  
  - University.Infrastructure: DbContext, Repositories (implementations), EF Configurations  
**Prevents:** Circular dependencies, persistence logic in domain  
**Rule:** No project may reference "lower" than itself; dependency flow: Web → App → Domain ← Infra  
**Status:** [ADOPTED] — Projects exist but empty; Week 1 fills them

### AD-10: Validation Strategy — FluentValidation
**Type:** decision  
**Binds:** Every Command handler has a paired Validator; MediatR pipeline executes validation before handler  
**Prevents:** Invalid state in domain, scattered null checks  
**Rule:** Validators run in middleware; if invalid, return failure without executing handler  
**Scope MVP:** Basic field validation (required, format); domain rules in entities  
**Status:** [NEW] — To implement Week 2

### AD-11: Testing — Unit + Integration (xUnit, EF InMemory)
**Type:** decision  
**Binds:** University.UnitTests (EF InMemory for domain/app logic), University.IntegrationTests (real DbContext instance)  
**Prevents:** No test coverage on critical paths, hard to refactor  
**Rule:** Domain entities tested in isolation; handlers tested with in-memory context; critical user flows end-to-end  
**Target Coverage:** 70% domain + application layers  
**Status:** [NEW] — To implement Weeks 2–4

### AD-12: Deployment — Docker Compose (Local MVP)
**Type:** decision  
**Binds:** docker-compose.yml with mssql + nehrportal services; single command spin-up  
**Prevents:** "It works on my machine" variance, manual DB setup  
**Rule:** DB migrations run at app startup; seed data inserted on first launch; subsequent restarts preserve data in volume  
**Graduation Demo:** `docker-compose up` → app ready in < 2 min  
**Status:** [NEW] — To implement Week 4

### AD-13: State Mutation — Immutability in Commands, Events in Domain
**Type:** decision  
**Binds:** Domain entities are mutable; state changes recorded via events (future Phase 2 audit log)  
**Prevents:** Hidden state changes, hard-to-track side effects  
**Rule:** Commands validate input before calling domain methods; domain entities mutate state; handlers publish events (Phase 2)  
**Scope MVP:** Events not yet consumed; infrastructure ready for Phase 2  
**Status:** [ADOPTED] — Deferred consumption until Phase 2

### AD-14: Error Handling — Result<T> Pattern + Application Exceptions
**Type:** decision  
**Binds:** Domain throws DomainException; handlers catch and return Result<T> with error code  
**Prevents:** Unhandled exceptions in UI, inconsistent error messages  
**Rule:** Try-catch at handler level; return structured error response to Blazor component  
**Scope MVP:** Basic error codes (ValidationError, NotFound, Unauthorized); detailed logging via ILogger  
**Status:** [NEW] — To implement Week 2

### AD-15: Logging — ILogger (Microsoft.Extensions.Logging)
**Type:** decision  
**Binds:** Injected ILogger<T> in handlers and repositories; logs all auth, data mutations, errors  
**Prevents:** Silent failures, hard-to-diagnose issues  
**Rule:** Info-level: user actions (login, submission); Debug-level: internal flow; Error-level: exceptions  
**Scope MVP:** Console + file sink locally; Phase 2 adds Application Insights  
**Status:** [NEW] — To implement Week 2

### AD-16: Session Management — ASP.NET Core Sessions + Claims
**Type:** decision  
**Binds:** HttpContext.User claims carry role/identity; session cookie 30-min timeout; logout clears cookie  
**Prevents:** Data leakage, credential theft, session fixation  
**Rule:** Middleware enforces auth on protected routes; MainLayout checks HttpContext.User.Identity.IsAuthenticated  
**Scope MVP:** No remember-me; Phase 2 considers device/location tracking  
**Status:** [NEW] — To implement Week 2

### AD-17: Caching — Minimal in MVP
**Type:** decision  
**Binds:** No distributed cache (Phase 2); EF Core query cache at DbContext level only  
**Prevents:** Stale data issues at graduation demo, complexity before needed  
**Rule:** Queries hit database; let SQL Server's buffer pool handle caching; Phase 2 evaluates Redis/MemoryCache per metrics  
**Status:** [NEW] — Deferred

### AD-18: API vs. Renderer Architecture
**Type:** decision  
**Binds:** No REST API in MVP; Blazor components call MediatR handlers directly via Dependency Injection  
**Prevents:** Over-engineering, deployment complexity  
**Rule:** UI layer is Blazor Server; business logic in MediatR handlers; database in EF Core  
**Scope MVP:** Single deployment unit; Phase 2 may split API for mobile/partner integrations  
**Status:** [NEW] — To implement
